# Product Requirements Document
## CDR/IPDR Analysis and Investigation Platform

**Version:** 1.0
**Status:** Draft for Review
**Date:** July 15, 2026

---

## 1. Executive Summary

Call Detail Records (CDR) and Internet Protocol Detail Records (IPDR) are the two primary evidentiary data sources used in telecom-based criminal investigation, fraud detection, and network intelligence. Investigators and analysts today rely heavily on spreadsheet-based manual correlation, which does not scale to the volume, heterogeneity, and velocity of records generated by modern telecom and ISP infrastructure.

This PRD defines requirements for a **CDR/IPDR Analysis Platform** — a system that ingests heterogeneous CDR and IPDR formats from multiple telecom operators/ISPs, normalizes them into a unified schema, and applies network analysis, pattern detection, and visualization techniques to surface actionable intelligence (association networks, movement patterns, communication anomalies, and digital-physical correlation).

The document also identifies the **research gap** this platform addresses relative to existing tools and published literature, framed using standard empirical software engineering methodology.

---

## 2. Problem Statement

Law enforcement agencies (LEAs), fraud investigation units, and telecom security teams face the following recurring problems:

1. **Format fragmentation** — Each telecom operator/ISP exports CDR/IPDR in a different column layout, encoding, and field-naming convention (no enforced universal schema in most jurisdictions), forcing analysts to manually remap fields for every case.
2. **Volume vs. manual workflow mismatch** — A single case can involve millions of CDR/IPDR rows across multiple numbers/IPs and multiple operators; spreadsheet-based cross-referencing is error-prone and slow.
3. **Disconnected analysis** — CDR (voice/SMS metadata) and IPDR (data-session metadata: source/destination IP, port, APN, session duration) are typically analyzed in isolation, losing correlation value (e.g., a voice call immediately followed by a data session to a specific service).
4. **Limited network/link analysis** — Most existing tools produce basic call-frequency tables rather than graph-based social network analysis (centrality, community detection) that would reveal hierarchy and key actors in a criminal network.
5. **No standard for encrypted/NAT'd traffic** — Widespread use of CG-NAT by ISPs and encrypted app traffic (VoIP-over-data, messaging apps) means IPDR alone frequently cannot attribute a session to an individual user without additional correlation logic.
6. **Chain-of-custody and courtroom admissibility** — Analytical outputs must be reproducible, auditable, and exportable in a form usable as evidence.

## 3. Goals and Objectives

| Goal | Description |
|---|---|
| G1 | Ingest and normalize CDR/IPDR from heterogeneous operator formats into one canonical schema |
| G2 | Provide call/session network graph analysis (centrality, community detection) to identify key actors and clusters |
| G3 | Correlate CDR and IPDR records for the same subject/number to build a unified activity timeline |
| G4 | Support tower/cell-ID based location and movement-pattern reconstruction |
| G5 | Detect anomalies (burner-number patterns, sudden network changes, odd-hour activity clusters) |
| G6 | Produce court-admissible, auditable, exportable reports |
| G7 | Operate at scale (multi-million row datasets) with acceptable query latency |

**Non-goals (out of scope for v1):** real-time lawful-interception (live wiretap) capability; decryption of application-layer payloads; and cross-border data-sharing workflows.

## 4. Target Users / Personas

- **Cyber Crime Investigator** — needs quick link analysis and timeline reconstruction per case.
- **Forensic Analyst** — needs deep, reproducible, exportable technical analysis for court submission.
- **Fraud/Risk Analyst (Telecom/BFSI)** — needs anomaly detection at scale across large subscriber bases.
- **Case Supervisor** — needs dashboard-level summaries and network visualizations to allocate investigative resources.

---

## 5. Scope

### In Scope
- CDR ingestion: voice calls, SMS metadata (caller, callee, timestamp, duration, cell-ID/tower, IMEI/IMSI where available).
- IPDR ingestion: session start/end, source/destination IP:port, protocol, APN, data volume, NAT translation logs where provided by the ISP.
- Format adapters for major operator/ISP export layouts (configurable mapping, not hardcoded).
- Unified data model joining CDR + IPDR + tower location reference data.
- Graph-based link/network analysis (degree, betweenness, eigenvector centrality; community detection).
- Timeline and geolocation visualization.
- Anomaly/pattern detection module (rule-based + statistical).
- Case management, audit trail, and export (PDF/CSV/graph export) for evidentiary use.

### Out of Scope (v1)
- Live interception or real-time traffic capture.
- Payload-level decryption of encrypted app traffic.
- Predictive/behavioral profiling beyond documented, explainable statistical methods (to avoid unreviewable "black box" evidentiary claims).

## 6. Functional Requirements

| ID | Requirement | Priority |
|---|---|---|
| FR1 | System shall support configurable import adapters for CSV/Excel/fixed-width CDR/IPDR formats | Must |
| FR2 | System shall normalize disparate field names/formats into a canonical schema (subject ID, timestamp, duration, source, destination, cell/tower ID, IP, port, protocol) | Must |
| FR3 | System shall deduplicate and validate records (timestamp integrity, malformed rows quarantine) | Must |
| FR4 | System shall construct a call/session graph per case, with nodes = subjects/numbers/IPs and edges = interactions (weighted by frequency/duration) | Must |
| FR5 | System shall compute centrality measures (degree, betweenness, closeness, eigenvector/PageRank) on the case graph | Must |
| FR6 | System shall run community detection (modularity-based) to surface clusters/sub-groups | Should |
| FR7 | System shall correlate CDR and IPDR entries for the same subject into a unified chronological timeline | Must |
| FR8 | System shall map cell-tower IDs to geographic coordinates and visualize movement over time | Must |
| FR9 | System shall flag statistical anomalies (e.g., new contact bursts, odd-hour clustering, single-use/burner-pattern numbers, rapid IMEI/SIM swap) | Should |
| FR10 | System shall allow analysts to annotate, tag, and build a case narrative on top of the graph/timeline | Should |
| FR11 | System shall export findings as a formatted report (PDF), raw data (CSV), and graph file (GraphML/GEXF for Gephi) | Must |
| FR12 | System shall maintain an immutable audit log of all data imports, transformations, and analyst actions for chain-of-custody | Must |
| FR13 | System shall support role-based access control per case | Must |

## 7. Non-Functional Requirements

- **Scale:** support ingestion and interactive querying of 5M+ CDR/IPDR rows per case without query latency exceeding a few seconds for common operations.
- **Data integrity:** all transformations must be deterministic and re-runnable (no silent data loss).
- **Security:** encryption at rest and in transit; strict access control given the sensitivity of subscriber data.
- **Auditability:** every analytical output must be traceable to source rows.
- **Portability:** deployable on-premises (common LEA requirement) as well as in a controlled cloud environment.
- **Explainability:** anomaly/pattern flags must show the underlying rule or statistic, not an opaque score, to remain evidentiarily defensible.

## 8. High-Level Architecture

```
[Operator/ISP Export Files] 
        │
        ▼
[Format Adapter Layer] → configurable per-operator field mapping
        │
        ▼
[Normalization & Validation Engine] → canonical schema, dedup, quarantine
        │
        ▼
[Unified Data Store] (CDR + IPDR + Tower/Cell reference + Case metadata)
        │
        ├──► [Graph Engine] → centrality, community detection (igraph/NetworkX/Neo4j)
        ├──► [Timeline/Correlation Engine] → CDR-IPDR joins per subject
        ├──► [Geolocation Engine] → tower-ID → lat/long, movement paths
        └──► [Anomaly Detection Engine] → rule-based + statistical scoring
        │
        ▼
[Visualization & Case Management UI] → link graphs, timelines, maps, reports
        │
        ▼
[Export/Audit Layer] → PDF/CSV/GraphML, immutable audit trail
```

**Suggested stack:** backend services in Python (pandas/Polars for large-record processing, NetworkX/igraph for graph analytics, or Neo4j for persistent large-scale graph queries), a web frontend for case dashboards and visualizations, and a relational or document store for case/audit metadata with a graph database for the network layer.

## 9. Core Data Model (Canonical Schema, Simplified)

**CDR (canonical fields):** `subject_id`, `event_type` (voice/SMS), `caller`, `callee`, `start_time`, `duration`, `cell_id`, `imei`, `imsi`, `operator`.

**IPDR (canonical fields):** `subject_id`, `session_start`, `session_end`, `source_ip`, `source_port`, `dest_ip`, `dest_port`, `protocol`, `apn`, `data_volume_up`, `data_volume_down`, `nat_translation_ref`.

**Reference:** `cell_tower_map` (cell_id → lat/long, operator), `case_metadata`, `audit_log`.

## 10. Success Metrics (KPIs)

- Time-to-first-insight per case (target: reduce from days to hours).
- % of records successfully auto-mapped by format adapters without manual correction.
- Analyst-reported precision of anomaly flags (false-positive rate tracked per release).
- Number of key-actor identifications validated as correct in closed cases (post-hoc accuracy).
- Report generation time for a fully annotated case export.

## 11. Research Gap

This section frames the platform's contribution against the existing body of work and commercial tooling, following standard empirical software-engineering gap-analysis practice (problem → existing approaches → limitation → proposed contribution).

### 11.1 State of Existing Approaches
- **Commercial/LEA tools** (in-house CDR analyzers, spreadsheet macros, some proprietary link-analysis suites) largely focus on single-format CDR import and basic frequency/duration tables; IPDR support is inconsistent or absent.
- **Academic literature** on CDR analysis has concentrated on telecom churn prediction, mobility modeling (human mobility from CDR), and fraud detection using call-graph features — but rarely on **forensic, evidentiary-grade** correlation of CDR with IPDR.
- **Social network analysis (SNA) research** provides mature centrality and community-detection methods, but their application to *heterogeneous, multi-operator, noisy investigative datasets* (rather than clean academic mobility datasets like the "Reality Mining" or D4D Orange datasets) is underexplored.
- **IPDR-specific research** is sparse in the public literature, largely because IPDR formats and CG-NAT translation logs are operator-proprietary and rarely released for study — creating a data-availability barrier to reproducible research.

### 11.2 Identified Gaps

| Gap | Description |
|---|---|
| **Schema standardization** | No widely adopted, openly published canonical schema/ontology unifying CDR and IPDR fields across operators; most work assumes a single clean format. |
| **CDR–IPDR joint correlation** | Very limited published methodology for reliably correlating a CDR event with a temporally adjacent IPDR session for the same subject, especially under CG-NAT where multiple subscribers share a public IP. |
| **Benchmark datasets** | No public, ethically-cleared benchmark dataset of realistic multi-operator CDR/IPDR pairs exists for reproducible evaluation (unlike CV/NLP fields) — most work uses private, non-shareable law-enforcement data. |
| **Graph analysis under noisy/adversarial conditions** | Existing centrality/community-detection methods assume relatively complete, honest graphs; criminal networks deliberately use burner numbers, SIM-swapping, and short-lived contacts to defeat straightforward graph analysis — a gap in *adversarially-robust* network analysis methods for this domain. |
| **Explainability for evidentiary use** | ML/statistical anomaly detection literature optimizes for accuracy/AUC, not for the courtroom-required explainability of *why* a pattern was flagged; there is little SE/HCI research on designing investigator-facing explainable anomaly interfaces for telecom forensics specifically. |
| **Scalability benchmarks** | Limited empirical data on how graph algorithms (betweenness centrality in particular, which is computationally expensive) perform at real case scale (millions of records, tens of thousands of nodes) versus the small synthetic graphs typically used in SNA papers. |
| **Privacy-preserving analysis** | Sparse work on techniques that allow cross-operator or cross-agency correlation of CDR/IPDR without full data pooling (e.g., privacy-preserving record linkage), despite growing regulatory sensitivity around subscriber data. |

### 11.3 Proposed Contribution
This platform's research contribution is positioned at the intersection of **empirical software engineering, network science, and digital forensics**:
1. A configurable, published canonical schema for CDR/IPDR normalization, tested against real-world multi-operator export samples.
2. A documented CDR–IPDR correlation methodology (including handling of CG-NAT ambiguity) with measurable precision/recall against ground-truth case outcomes.
3. Empirical benchmarking of standard SNA algorithms (centrality, community detection) on realistic-scale, noisy investigative graphs, including performance under adversarial evasion patterns (burner numbers, SIM swapping).
4. An explainable-by-design anomaly detection layer, evaluated not just on detection accuracy but on investigator trust/usability (HCI-style evaluation), addressing the evidentiary-explainability gap.
5. A scalability evaluation of graph algorithms at real case-file scale, filling the reported gap between academic SNA benchmarks and operational data volumes.

### 11.4 Suggested Validation Methodology
Following empirical SE practice (see Section 12 references): a **case-study-based evaluation** using anonymized/synthetic case data across multiple simulated operator formats, triangulated with **controlled-experiment style A/B testing** of investigator task completion time and accuracy with vs. without the platform, and a **mining-software-repositories-style** log analysis of platform usage patterns once deployed, to iteratively refine which features actually drive investigative outcomes.

## 12. Risks and Assumptions

- **Assumption:** Operators/ISPs will provide IPDR including NAT translation logs where legally mandated; without this, IP-to-subscriber attribution accuracy drops significantly.
- **Risk:** Legal/regulatory constraints on data retention and cross-jurisdiction sharing may limit dataset availability for both the product and any accompanying research validation.
- **Risk:** Betweenness centrality and similar algorithms are computationally expensive at scale; architecture must budget for approximate/streaming graph algorithms as a fallback.
- **Risk:** Over-reliance on anomaly flags without explainability could produce evidentiary challenges in court; mitigated by FR9/explainability requirement.
